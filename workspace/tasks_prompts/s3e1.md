## Mission

Find all anomalies in power plant sensor data and submit their file IDs.

## Data

Download sensor readings from `https://hub.ag3nts.org/dane/sensors.zip` — approximately 10,000 JSON files.

After downloading, unzip and inspect a few sample files to confirm the format before writing any analysis code.

## Specification

Each JSON file contains a single sensor reading in this format:

```json
{
  "sensor_type": "temperature/voltage",
  "timestamp": 1774064280,
  "temperature_K": 612,
  "pressure_bar": 0,
  "water_level_meters": 0,
  "voltage_supply_v": 230.4,
  "humidity_percent": 0,
  "operator_notes": "Readings look stable and within expected range."
}
```

### Fields

- **sensor_type** — active sensor name or set of sensors separated by `/` (e.g. `temperature`, `water`, `voltage/temperature`)
- **timestamp** — Unix timestamp
- **temperature_K** — temperature in Kelvin
- **pressure_bar** — pressure in bars
- **water_level_meters** — water level in meters
- **voltage_supply_v** — supply voltage in V
- **humidity_percent** — humidity in percent
- **operator_notes** — operator note in English

All measurement fields are always present. For inactive sensors the value must be **0**.

Integrated sensors (2–3 types combined) return all fields defined by their component sensors.

### Valid ranges for active sensors

- **temperature_K**: 553 – 873
- **pressure_bar**: 60 – 160
- **water_level_meters**: 5.0 – 15.0
- **voltage_supply_v**: 229.0 – 231.0
- **humidity_percent**: 40.0 – 80.0

### Anomaly definitions

- Measurement data is outside valid ranges
- Operator claims everything is OK, but data is incorrect
- Operator claims errors were found, but data is actually OK
- Sensor returns data it should not (e.g. a water level sensor reporting voltage)

## Strategy

There are ~10,000 files. Sending them all to an LLM would be extremely expensive and wasteful. Use the think tool to plan a multi-pass approach:

1. **Programmatic pass** — Most anomaly types are fully deterministic (out-of-range values, sensors reporting fields they shouldn't, inactive fields with non-zero values). Write code to check every file against the rules above. This should catch the majority of anomalies without any LLM calls.

2. **Narrowing** — After the programmatic pass, you'll have two sets: files already flagged as anomalous, and files that passed all numeric/structural checks. Only the second set needs further analysis.

3. **Semantic pass** — For files that passed programmatic checks, the remaining anomaly type involves operator notes that contradict the actual sensor data. This is where LLM is needed — but only on the filtered subset, not all 10,000 files. Think about how to batch this efficiently.

After each pass, use the think tool to review your results before moving on. Are the numbers plausible? Did you cover all anomaly types from the spec?
