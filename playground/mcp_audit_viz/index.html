MCP Handling Architecture

Our implementation (aidevs4) vs. Wonderlands reference — CLI + Hono server + Slack bot — what, why, and how to close the gaps

The Big Picture

What is MCP?

MCP (Model Context Protocol) lets your agent talk to external tool servers. Instead of coding every tool in TypeScript, you connect to MCP servers that expose tools over a standard protocol. Think of it like USB for AI tools — plug in a server, and the agent can use its tools.

Each MCP server can expose many tools (e.g., a "gmail" server might expose send_email, search_inbox, read_thread). Your agent discovers these tools at startup and uses them like any other tool.

Our agent runs in 3 deployment modes — CLI (src/cli.ts), Hono HTTP server (src/server.ts), and Slack bot (src/slack.ts). All three share the exact same MCP initialization via bootstrap.initServices(). This means MCP tools exposed to the CLI are equally exposed to Slack users and HTTP API callers. The threat model is NOT "local CLI" — it's "network-accessible service".

Threat Model — Three Entry Points, One MCP Pool

Why this changes everything

The previous version of this audit assumed a local CLI-only deployment. In reality, our agent has three entry points with very different trust boundaries. All three share a single bootstrap.initServices() call that connects all MCP servers and registers all tools once — there is no per-request or per-mode isolation.

This means a Slack user typing @agent send an email triggers the same mcp_gmail_send tool as the developer running bun run agent locally. The blast radius of every MCP gap scales to the least-trusted entry point.

What this means for MCP safety Critical
  • Size limits are now CRITICAL, not just nice-to-have. A Slack message triggering a tool that returns 5MB of data could blow up the context for all concurrent users.
  • Error classification matters more — Slack users see confusing generic errors instead of "server timed out, retrying".
  • Server status prevents wasted LLM calls — don't send a tool call to a degraded server just because it's in the Map.
  • OAuth tokens stored as plaintext are now accessible to any process on the server, not just the local developer. Consider encryption.
  • The sampling handler (where MCP servers can ask our LLM to generate text) is now invocable by any MCP server triggered by any Slack user — this is a cost amplification vector.
Wonderlands addresses this with Reference
  • Per-tenant scoping — each tenant only sees their MCP servers and tools
  • Tool profile assignments — tools are explicitly assigned to profiles, not globally available
  • Confirmation gates with fingerprint approval — destructive tools require human confirmation per-schema-version
  • Token encryption at rest with MCP_SECRET_ENCRYPTION_KEY
  • In-flight correlation — tracks which tenant/run triggered which MCP call
  • allowedTenantIds whitelist on static servers — not all servers available to all tenants
We don't need multi-tenancy, but we DO need: (1) output size limits to protect shared context, (2) per-entry-point tool filtering (e.g. Slack shouldn't access filesystem tools), and (3) token encryption since this isn't just a local CLI anymore.

Server Lifecycle

How does a connection live and die?

An MCP server connection goes through distinct phases: load config → create transport → connect client → discover tools → ready for calls → disconnect. Each phase can fail, and how we handle those failures determines reliability.

Connection lifecycle — sequence diagram
Our Lifecycle Current
  • Binary state: a server is either in the servers Map or it's not
  • If connect() fails, server is silently skipped (warning logged)
  • If listTools() fails, no tools registered but server stays "connected"
  • No way to query server health after startup
  • Hot-reload support via global __mcpService
Wonderlands Lifecycle Reference
  • 4-state model: connecting → ready | degraded | authorization_required
  • Each state carries lastError for diagnostics
  • Status checked before every tool call — no calls to degraded servers
  • Paginated tool discovery (cursor-based) for large tool sets
  • Explicit refreshServer() for reconnection
  • Persists discovery state and errors to DB

Tool Execution Flow

What happens when the LLM calls an MCP tool?

When the agent's LLM output says "call mcp_gmail_send with args {...}", the runtime needs to: find the right MCP server, forward the call, enforce a timeout, validate the response, normalize the content format, and return it to the LLM. Each of these steps is a place where things can go wrong.

Tool call pipeline — ours vs. ideal

The key insight: untrusted boundary

MCP servers are external processes. They might be buggy, compromised, or just poorly written. The response they send is like user input — you must validate it before trusting it. Our current code trusts everything: no size checks, no schema validation, no type guards. The ideal pipeline treats the response as adversarial.

Think of it like an HTTP API: you wouldn't take a random JSON response and pass it straight to your frontend without validation. Same principle applies here — the MCP server is the API, and our agent is the consumer.

Safety & Size Limits

Why size limits matter

LLMs have a context window (a maximum amount of text they can process at once). Every character in a tool response eats into that budget. If an MCP server returns 5MB of text, it will either: (a) blow the context limit and crash, (b) cost a fortune in tokens, or (c) push important earlier context out of the window. Size limits are not optional — they're a circuit breaker.

Our mapMcpContent() No limits
// src/infra/mcp.ts:57
function mapMcpContent(content: unknown[]): ContentPart[] {
  return (content as Record<string, unknown>[])
    .map((item): ContentPart => {
      const type = item.type as string;
      if (type === "text") {
        return { type: "text", text: item.text as string };
        //                      ^^^^^^^^^^^^^^^^^^^^^^^^^^
        // No size check! Could be megabytes.
      }
      if (type === "image") {
        return {
          type: "image",
          data: item.data as string,  // No size check!
          mimeType: item.mimeType as string,
        };
      }
      // ... resource handling ...
      return { type: "text", text: JSON.stringify(item) };
      //  Unknown type → JSON stringify, also no limit
    });
}
Every content block passes through with zero size enforcement. A single tool call could return a 50MB response and it would be forwarded to the LLM verbatim.
Wonderlands normalize-result.ts With limits
// adapters/mcp/normalize-result.ts
const MAX_RESOURCE_TEXT_BYTES = 32_768;   // 32KB per block
const MAX_TOOL_OUTPUT_BYTES  = 131_072;   // 128KB total

const normalizeContentBlock = (block) => {
  switch (block.type) {
    case 'text':
      return ok({ text: block.text, type: 'text' });

    case 'resource':
      if ('blob' in block.resource) {
        return err({ message: 'inline binary not supported',
                     type: 'validation' });
      }
      if (Buffer.byteLength(block.resource.text, 'utf8')
            > MAX_RESOURCE_TEXT_BYTES) {
        return err({ message: 'resource exceeds size limit',
                     type: 'capacity' });
      }
      return ok({ resource: { ... }, type: 'resource' });

    case 'audio':
    case 'image':
      return err({ message: `unsupported ${block.type}`,
                   type: 'validation' });
  }
};

// After normalizing all blocks, check total:
if (Buffer.byteLength(JSON.stringify(output), 'utf8')
      > MAX_TOOL_OUTPUT_BYTES) {
  return err({ message: 'result exceeds output limit',
               type: 'capacity' });
}
Two-layer defense: per-block limit (32KB) catches individual huge items, total limit (128KB) catches many-small-items attacks. Binary content is rejected outright since it's not useful for text-based LLMs.

Error Handling

Why error classification matters

When a tool call fails, the agent needs to decide: should I retry? With our current code, every error looks the same — a generic string. The agent (and the user) can't tell if the server timed out (retry makes sense), if auth expired (need to re-authorize), or if the arguments were wrong (retry is pointless). The Wonderlands approach classifies errors into distinct types so each can be handled appropriately.

Our Error Handling All errors identical
// src/infra/mcp.ts:319-321
} catch (err) {

    `Error calling MCP tool "${tool.name}"` +

Every error — timeout, auth failure, invalid params, server crash — becomes the same toolError(string). The LLM sees "Error calling MCP tool X: some message" and has to guess what went wrong from the message text.

Wonderlands Error Handling Classified
// adapters/mcp/errors.ts
export const toMcpDomainError = (serverId, error) => {
  // HTTP-level errors
  if (error instanceof StreamableHTTPError) {
    if (error.code === 408 || error.code === 504)
      return { type: 'timeout', message: '...' };
    return { type: 'provider', message: '...' };
  }

  // MCP protocol errors
  if (error instanceof McpError) {
    switch (error.code) {
      case ErrorCode.RequestTimeout:
        return { type: 'timeout', ... };
      case ErrorCode.InvalidParams:
        return { type: 'validation', ... };
      case ErrorCode.MethodNotFound:
        return { type: 'not_found', ... };
      default:
        return { type: 'provider', ... };
    }
  }

  // Generic JS errors
  if (error instanceof Error) {
    if (/timed? out/i.test(error.message))
      return { type: 'timeout', ... };
    return { type: 'provider', ... };
  }

  return { type: 'provider', message: 'Unknown' };
};

OAuth Flow

How do MCP servers authenticate?

Some MCP servers (like Gmail) need OAuth to access user data. The flow: our agent tries to connect → server says "unauthorized" → we open a browser for the user to log in → browser redirects back to our local callback server with an auth code → we exchange the code for tokens → reconnect. Tokens are saved to disk so you only authenticate once.

Since our agent runs as a Hono server and Slack bot (not just CLI), OAuth tokens stored as plaintext JSON on the server filesystem are a real concern. Any process on that machine, or any vulnerability that allows file reads, exposes access tokens to Gmail and other services. Wonderlands encrypts tokens at rest with MCP_SECRET_ENCRYPTION_KEY — we should do the same.
OAuth authorization code flow (our implementation)
Our OAuth Needs encryption

  • PKCE support (code verifier saved to disk)
  • Browser auto-open (macOS/Linux/Windows)
  • 5-minute timeout on callback

  • Single auth mode: Authorization Code

Wonderlands OAuth Production-grade

  • 5 auth modes: bearer, auth code, client credentials, private key JWT, static JWT
  • Per-tenant/per-account credential scoping
  • 15-minute TTL on authorization state (auto-cleanup)
  • Discovery state caching
  • Token check before connect — skip OAuth if tokens exist

Server Status Model

Why a status model?

Currently, we track servers with a binary flag: either they're in the servers Map (connected) or not. This means we can't distinguish between "server connected but tool discovery failed" and "server fully ready". The Wonderlands model uses 4 distinct states with transitions, making failures visible and actionable.

What NOT to copy from Wonderlands

Not everything in the reference applies to our multi-mode agent:

  • Multi-tenant scoping (tenantId, accountId) — we're single-operator, multiple entry points
  • Database persistence of server definitions — config file is fine
  • HTTP API routes for server CRUD — we manage via mcp.json
  • 5 OAuth modes — auth code + bearer is sufficient

What we DO need (updated for server/Slack reality)

  • Token encryption at rest — this IS a server now, not just local CLI
  • Per-entry-point tool allow-lists — Slack should NOT access filesystem tools
  • Confirmation gates for destructive MCP tools (especially from Slack/HTTP)
  • Sampling handler cost controls — MCP servers can trigger LLM calls via any entry point
The previous version said "skip token encryption, filesystem IS the trust boundary." That was wrong — our agent is a network-accessible service. Every safety gap scales to the least-trusted entry point (Slack).
