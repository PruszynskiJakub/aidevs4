Scenario: Agent needs user approval for a destructive action

Run is executing Running
Agent is in Turn N, LLM responded with tool calls.
LLM requests: send_email(...) Tool Call
Irreversible action. Tool registered with meta.confirmIf: () => true.
confirmBatch(calls) Gate
For each call: parse args via safeParse, evaluate confirmIf({ action, args, toolCallId }). send_email needs approval → cached in pendingConfirmations with new confirmationId.
return GateResult.waitingOn Signal
SP-92: No exception. confirmBatch returns { approved: [...auto], denied: [], waitingOn: { kind: "user_approval", confirmationId, prompt } }.
dispatchTools bubbles waitingOn up. runCycle returns { kind: "waiting", waitingOn }. runIteration returns RunExit { kind: "waiting", waitingOn }.
persistRunExit() → run.waiting Persisted
updateRunStatus(runId, { status: "waiting", waitingOn: JSON.stringify(...) }).
bus.emit("run.waiting", { waitingOn }). Loop exits cleanly. Process can shut down safely — all state in SQLite.
WAITING FOR HUMAN — could be seconds, hours, or days
Human provides decision Resolve
Surface: CLI provider, HTTP, Slack (slack-confirmation.ts). Resolution: { kind: "user_approval", confirmationId, decisions: { toolCallId: "approve" | "deny" } }.
resumeRun(runId, resolution) Entry
resume-run.ts. Idempotent: if status !== "waiting", returns current RunExit as no-op. Asserts resolution.kind === waitingOn.kind.
findPendingToolCalls(messages) Locate
Walks transcript backwards to the most recent assistant message whose toolCalls still lack role:"tool" responses.
Dispatch approved / synthesize denied Execute
For each pending call: "approve" → dispatch(name, args, callId). "deny" (or missing) → synthetic "Error: Tool call denied by operator.".
Append synthetic tool outputs Persist
sessionService.appendRun() writes results as items. updateRunStatus({ status: "running", waitingOn: null, expectedVersion }) — optimistic lock guards double-resume.
run.resumed Event
bus.emit("run.resumed", { resolution }). RunState rehydrated from DB → runAndPersist(state) re-enters the loop.
Loop continues from Turn N+1 Continue
LLM sees the tool results (approved ran, denied carry rejection text). Proceeds normally toward the next terminal exit.

Key Design Properties

Crash-safe
State persisted to SQLite (runs, items) before waiting. reconcileOrphanedWaits() runs at startup from bootstrap.ts and resumes parents whose children already finished.
Idempotent resume
Optimistic locking via version column — updateRunStatus({ expectedVersion }) rejects stale writes. Re-calling resumeRun on a non-waiting run returns the current RunExit as a no-op.
Return-value signaling
SP-92: no thrown WaitRequested. Waiting flows back through GateResult → CycleOutcome → RunExit. The compiler enforces handling at every level.
Multi-surface
setConfirmationProvider() swaps the surface: CLI inline, HTTP API, or slack-confirmation.ts. The wait/resume protocol is transport-agnostic.

Multi-Agent Delegation with Nested Human-in-the-Loop

Two-step delegation in orchestrator.ts:
  1. createChildRun(opts) — fresh sessionId, insert run row (parentId, shared rootRunId), append user prompt. Does not enter the loop.
  2. Tool returns { wait: { kind: "child_run", childRunId } }; loop returns RunExit.waiting; persistRunExit() parks the parent; kickChildRunAsync() fires-and-forgets startChildRun(childRunId).
  3. startChildRun hydrates the row to running and re-enters runAndPersist for the child.
  4. Child terminal → run-continuation.ts subscriber catches run.completed/run.failed, calls findRunWaitingOnChild(childRunId), emits run.child_terminal, then resumeRun(parent, { kind: "child_run", result }).
